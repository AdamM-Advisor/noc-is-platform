## 2. DUCKDB CONNECTION MANAGER (`database.py`)

Implementasikan DuckDB connection manager dengan spesifikasi:

    KRITIS — DuckDB single-writer constraint:
    • Hanya 1 write connection boleh aktif sekaligus
    • Multiple read connections boleh paralel
    • Gunakan threading.Lock() untuk write operations
    • SELALU set memory_limit dan threads saat connect
    
    Pattern:
      get_connection()    → Read-only connection (bisa banyak)
      get_write_connection() → Write connection dengan lock (hanya 1)
    
      Setiap connection HARUS set:
        SET memory_limit='{DUCKDB_MEMORY_LIMIT}';
        SET threads={DUCKDB_THREADS};
    
      Connection HARUS di-close setelah digunakan.
      Gunakan context manager pattern.

## 3. CHUNKED UPLOAD (`upload.py` + `upload_service.py` + `UploadPage.jsx`)

### Backend Endpoints:

    POST /api/upload/single
      ← File < 10 MB langsung upload
      → Simpan ke UPLOAD_DIR, return {filename, size, path}
    
    POST /api/upload/chunk
      ← Headers: X-Upload-Id, X-Chunk-Index, X-Total-Chunks, X-Filename
      ← Body: binary chunk (max 5 MB)
      → Simpan chunk ke CHUNK_DIR/{upload_id}/chunk_{index}
    
    POST /api/upload/chunk/complete
      ← Body: {upload_id, filename, total_chunks}
      → Reassemble chunks → simpan ke UPLOAD_DIR
      → Hapus temp chunks
      → Return {filename, size, path}
    
    GET /api/upload/chunk/status/{upload_id}
      → Return {received_chunks, total_chunks, is_complete}

### Frontend Upload UI (`UploadPage.jsx`):

    UI Requirements:
    1. Drag-and-drop zone + file picker button
    2. Support file types: .xlsx, .csv, .parquet
    3. Auto-detect: jika file < 10MB → single upload
                     jika file ≥ 10MB → chunked upload (5MB per chunk)
    4. Progress bar yang menunjukkan:
       - Single: upload percentage
       - Chunked: "Chunk 3/8 (37.5%)" + overall percentage bar
    5. Status messages:
       - "Mengunggah..." → "Memproses..." → "Selesai ✓" / "Gagal ✗"
    6. Retry button jika gagal (per chunk untuk chunked upload)
    7. Upload history list (session, belum perlu persist)
    
    CATATAN: Di prompt ini, upload hanya menyimpan file.
    Processing (parsing, insert ke DB) akan ditambahkan di Prompt 2.

## 4. AUTO-BACKUP MECHANISM (`backup_service.py` + `admin.py`)

    TRIGGER BACKUP:
      • Otomatis SEBELUM setiap import data (akan dipanggil dari Prompt 2)
      • Manual dari Settings page (tombol "Backup Sekarang")
    
    MEKANISME:
      1. Copy file data/noc_analytics.duckdb 
         → data/backups/noc_analytics_{YYYYMMDD_HHMMSS}.duckdb
      2. Jika jumlah backup > MAX_BACKUP_COUNT (3):
         → Hapus backup TERLAMA
      3. Return: {backup_path, size_mb, timestamp, total_backups}
    
    RESTORE:
      POST /api/admin/restore
      ← Body: {backup_filename}
      → Copy backup → timpa database utama
      → Restart DuckDB connection
      → Return: {restored_from, timestamp}
    
    ENDPOINTS:
      GET  /api/admin/backups        → List semua backup {name, size, date}
      POST /api/admin/backup         → Trigger manual backup
      POST /api/admin/restore        → Restore dari backup tertentu
      GET  /api/admin/db-info        → {db_size_mb, table_count, row_counts}
